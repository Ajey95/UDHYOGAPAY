\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

UdhyogaPay is a full-stack hyperlocal marketplace platform that bridges the gap between skilled service workers and users who require on-demand professional services such as plumbing, electrical repair, carpentry, domestic cleaning, and related trades. The platform employs a modern technology stack comprising React 19 with TypeScript on the frontend, Node.js and Express on the backend, MongoDB as the primary NoSQL database, Socket.io for real-time bidirectional communication, Leaflet.js with OpenStreetMap for interactive geolocation discovery, and Cloudinary for cloud-based media management.

The core innovation of UdhyogaPay lies in its AI-assisted multi-factor worker ranking algorithm that computes a composite match score using distance proximity (50\%), average rating (30\%), and years of experience (20\%), augmented by an urgency multiplier. This ensures that the most suitable worker appears at the top of search results rather than merely the nearest one. The platform is secured end-to-end using JWT tokens stored in httpOnly cookies, bcrypt password hashing with a cost factor of 12, rate limiting, Helmet.js HTTP security headers, and rigorous MongoDB aggregation injection prevention.

The system supports three distinct roles: authenticated \textbf{Users} who can discover, book, and pay for services; \textbf{Workers} who undergo KYC verification and manage their availability and earnings; and \textbf{Administrators} who oversee the entire platform including worker approvals, complaint management, and financial payouts. The platform was built and tested over one academic year, achieving full deployment readiness on Railway (backend), Netlify (frontend), and MongoDB Atlas (database).

\vspace{0.5cm}
\noindent\textbf{Keywords:} Hyperlocal Marketplace, Service Booking, Geolocation Matching, Full-Stack Web Application, JWT Authentication, MongoDB, Socket.io, React TypeScript.

\section{Background and Motivation}

The unorganised service sector in India employs more than 120 million workers spanning trades such as plumbing, electrical fitting, carpentry, and domestic help. Despite this enormous workforce, both workers and customers suffer from severe information asymmetry: customers have no reliable way to find verified, nearby workers; workers have no structured channel to market their skills, maintain a reputation, or receive digital payments. Word-of-mouth remains the dominant discovery mechanism, and cash is almost universally the only payment channel, leaving both parties without legal recourse when disputes arise.

Platforms such as Urban Company (erstwhile UrbanClap) demonstrated that aggregating and verifying local tradespeople into a single digital marketplace creates enormous value. However, Urban Company's model is predominantly suited to metropolitan India and requires workers to undergo lengthy in-person onboarding. There is a clear gap for a lightweight, mobile-friendly, geolocation-first platform that any semi-urban or rural worker can join using a smartphone, receive bookings in real-time, and get paid digitally.

UdhyogaPay was conceived to address this gap. The name is a portmanteau of \textit{Udhyoga} (meaning occupation or work in Sanskrit and several South Indian languages) and \textit{Pay}, reflecting the dual focus on work discovery and digital payment. The platform is designed so that a first-time worker can register, upload KYC documents, and start receiving bookings within minutes of approval by an administrator.

\section{Problem Statement}

Despite the widespread adoption of the internet and smartphones across India, the local service economy remains largely undigitised. The key problems are as follows. Customers searching for a plumber or electrician in a specific locality have no way to see a verified list of available workers, compare ratings, or book an appointment without resorting to neighbour referrals or physical notice boards. Workers, on the other hand, have no online presence and cannot build a portable digital reputation that travels with them across cities. Payment disputes between customers and workers are common and have no recourse because all transactions happen in cash outside any audit trail. There is no platform that simultaneously handles geolocation-based discovery, real-time booking status updates, digital payment tracking, and worker KYC verification in a single unified interface tailored to the Indian semi-urban context.

\section{Objectives}

The primary objectives of this project are as follows. First, to design and implement a production-grade full-stack web application that connects service workers with users in a hyper-localised geographic area. Second, to build a multi-factor worker ranking algorithm that returns search results ordered by composite relevance rather than by distance alone. Third, to provide a complete digital audit trail for every booking and payment, thus eliminating the cash-only dispute problem. Fourth, to implement role-based access control across three user roles (User, Worker, and Admin) with proper JWT-based authentication. Fifth, to demonstrate real-time booking status propagation using WebSockets so that both parties always see the current state of a booking without polling. Sixth, to implement a rigorous KYC verification pipeline so that only verified, trustworthy workers appear in search results. Seventh, to deploy the entire platform on free-tier cloud infrastructure so that it is accessible publicly for demonstration and further development.

\section{Scope of the Project}

The scope covers the web-based version of the platform, which targets both desktop and mobile browsers. The mobile application built using React Native and Expo is outside the primary scope of this report but shares the same backend API. The platform covers six service categories as seed data: Plumbing, Electrical Work, Carpentry, Domestic Cleaning, Painting, and Air-Conditioner Servicing. Payment integration is architected and code-complete but operates in demo mode because live Razorpay API keys require a registered business entity. All other features — discovery, booking, KYC, real-time updates, reviews, and admin management — are fully operational and tested.

\section{Overview of Existing Platforms}

The hyperlocal service marketplace space has been explored commercially by several major players. Understanding their architectures, business models, and shortcomings is essential for positioning UdhyogaPay effectively.

\subsection{Urban Company (UrbanClap)}

Urban Company, rebranded from UrbanClap in 2020, is the market leader in the organised home services space in India. Founded in 2014, the platform employs a two-sided marketplace model where customers can discover, book, and pay for services ranging from beauty treatments to appliance repair. Urban Company employs a proprietary matching algorithm that considers service type, location, availability, and ratings. Workers are employed on a partner model, receive training and a branded kit, and must accept platform pricing. The platform uses React Native for its mobile apps and a microservices backend on AWS.

The limitation relevant to this work is that Urban Company's partner model requires significant capital investment from the company for worker onboarding, making it inherently metropolitan-centric. Semi-urban workers cannot self-register.

\subsection{TaskRabbit}

TaskRabbit, now a subsidiary of IKEA, operates in the USA, the UK, and selected European markets. It is a pure peer-to-peer marketplace where skilled individuals (Taskers) set their own hourly rates and bid on posted tasks. The platform uses a background-check and identity-verification pipeline provided by Checkr. TaskRabbit's model is ideal for markets with robust credit infrastructure but is unsuitable for India where credit card penetration is low and formal background checks are expensive.

\subsection{Sulekha}

Sulekha is a lead generation platform rather than a full marketplace. It connects customers with service providers by forwarding customer contact information to multiple businesses. There is no booking management, no escrow payment, and no real-time tracking. Workers often receive irrelevant leads and must filter waste contacts. Sulekha's model predates the smartphone era and has not fundamentally evolved.

\subsection{UrbanPro and NoBroker}

UrbanPro focuses on education and tutoring with a lead-generation model similar to Sulekha. NoBroker, while primarily a real-estate platform, has recently launched a home services vertical (NoBroker Hood) that is tightly coupled to its apartment management system, making it inaccessible to standalone service workers.

\section{Research Gaps Identified}

From the survey above, three clear research gaps emerge. First, no existing platform provides a fully self-service worker registration pipeline that operates without manual city-level relationship managers, making expansion to tier-2 and tier-3 cities economically unviable for competitors. Second, existing matching algorithms are proprietary and unpublished; there is academic value in implementing and documenting a transparent, weighted multi-factor scoring formula that can be independently verified and improved. Third, no publicly available open-source codebase demonstrates the full integration of geolocation search, real-time WebSocket booking updates, and an admin KYC pipeline in a single deployable project, creating a barrier for researchers and startups looking to build on prior work.

UdhyogaPay addresses all three gaps: it offers fully self-service registration, a documented and open-source matching algorithm, and a complete deployable reference architecture.

\section{Academic Literature}

The academic literature relevant to this project spans several areas. Geospatial query optimisation literature, particularly regarding MongoDB's \texttt{\$near} and \texttt{2dsphere} indexing, documents query response times of under 10 milliseconds for collections of up to one million documents when properly indexed \cite{mongo2dsphere}. Work on reputation systems in two-sided marketplaces \cite{reputationsystems} demonstrates that multi-factor reputation scores (combining recency, volume, and variance of ratings) produce better marketplace equilibria than simple average ratings. Research on WebSocket scalability \cite{websocketscale} shows that Socket.io with Redis adapter can handle over 10,000 concurrent connections per server instance, validating the technology choice for real-time booking updates at scale.

\section{High-Level Architecture}

UdhyogaPay follows a classic three-tier web architecture augmented with a separate AI Matching Engine microservice. Figure~\ref{fig:arch} (described below) shows the overall system topology.

The \textbf{Presentation Tier} consists of the React 19 / TypeScript single-page application served as static files from Netlify's CDN. The SPA communicates with the backend exclusively through REST HTTP calls and a persistent Socket.io WebSocket connection.

The \textbf{Application Tier} consists of two Node.js services. The primary \textbf{Backend API} (Express.js, port 5000) handles authentication, user/worker management, booking lifecycle, payment recording, and Socket.io event emission. The secondary \textbf{AI Matching Engine} (Express.js, port 5002) exposes a single \texttt{POST /match} endpoint that receives a search payload and returns a ranked list of workers. The backend calls the AI engine via an internal HTTP request and falls back to a Haversine distance-only ranking if the engine is unavailable.

The \textbf{Data Tier} uses MongoDB Atlas (cloud-hosted) as the primary data store, with Cloudinary providing an external object store for images (profile photos, KYC documents, service completion photos).

\section{Module Decomposition}

The system is decomposed into seven functional modules.

\subsection{Authentication Module}

This module handles all operations related to identity: registration, login, logout, password change, and password reset. It issues and validates JWT tokens and enforces role-based access control across all other modules. The key files are \texttt{authController.ts}, \texttt{authRoutes.ts}, and \texttt{auth.ts} middleware.

\subsection{User Module}

This module serves authenticated customers. It manages user profiles, saved addresses (with a maximum of five per user), the worker discovery and search flow, booking creation, and review submission. The profile update flow supports location update via the browser's Geolocation API.

\subsection{Worker Module}

This module manages the supply side of the marketplace. It covers the worker application with KYC document uploads, the admin approval pipeline, availability toggling (online/offline), booking acceptance and rejection, service status updates, earnings analytics, and payout requests.

\subsection{Booking Module}

This module governs the complete lifecycle of a service engagement from creation through to completion, payment, and review. It enforces state-machine transitions (pending → accepted → in-progress → completed or cancelled) and emits Socket.io events at each transition.

\subsection{Payment Module}

This module records all financial transactions. Each booking can have one associated transaction document. The module supports Razorpay online payment (with signature verification webhook), cash-on-delivery recording by either party, and admin-initiated refunds.

\subsection{Admin Module}

This module provides platform governance. It exposes endpoints and a React dashboard for worker KYC approval, user management, complaint resolution, review moderation, payout management, and an executive analytics dashboard backed by MongoDB aggregation pipelines.

\subsection{Real-time Module (Socket.io)}

This module manages the WebSocket layer. It handles connection authentication by extracting the JWT from the connection handshake, room assignment (each booking gets a dedicated Socket.io room), and event emission for booking status changes, payment confirmations, and chat messages.

\section{Backend — Controllers}

\subsection{authController.ts}

\textbf{Location:} \texttt{backend/src/controllers/authController.ts}

\textbf{Purpose:} Handles all authentication-related HTTP request/response cycles. It is responsible for user registration, login, logout, profile retrieval, password update, and the two-step password reset flow.

\textbf{Key Functions:}

\texttt{register()} receives the registration payload, validates that email and phone are unique via Mongoose's unique index constraint (catching the E11000 duplicate key error), hashes the password using \texttt{bcrypt.hash(password, 12)}, creates the User document, generates a signed JWT with \texttt{jsonwebtoken.sign(\{ id: user.\_id \}, secret, \{ expiresIn: '7d' \})}, and sets it as an httpOnly, sameSite=strict cookie. The function also creates a Worker stub document if the role is \texttt{worker}, so that the worker application form has a parent document to attach to.

\texttt{login()} finds the user by email using \texttt{User.findOne(\{ email \}).select('+password')}, compares the submitted password against the stored hash with \texttt{bcrypt.compare()}, and issues a fresh JWT on match. It returns a 401 with the opaque message ``Invalid credentials'' on any failure to prevent enumeration.

\texttt{forgotPassword()} generates a random 20-byte token using \texttt{crypto.randomBytes(20)}, stores its SHA-256 hash in the \texttt{resetPasswordToken} field with a 10-minute expiry, and sends the plain token to the user's email via Nodemailer. \texttt{resetPassword()} re-hashes the submitted token and queries for a user with matching \texttt{resetPasswordToken} and \texttt{resetPasswordExpire > Date.now()}.

\begin{lstlisting}[language=Java,caption={Register handler — token issuance}]
const token = jwt.sign(
  { id: user._id, role: user.role },
  process.env.JWT_SECRET!,
  { expiresIn: process.env.JWT_EXPIRE || '7d' }
);
res.cookie('token', token, {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'strict',
  maxAge: 7 * 24 * 60 * 60 * 1000
});
\end{lstlisting}

\subsection{bookingController.ts}

\textbf{Location:} \texttt{backend/src/controllers/bookingController.ts}

\textbf{Purpose:} Manages the complete booking lifecycle. It is the most complex controller in the system, handling creation, status transitions, cancellation, and the Socket.io event emission at each transition.

\textbf{Key Functions:}

\texttt{createBooking()} validates that the target worker is online and verified, creates the Booking document with status \texttt{pending}, saves the job site coordinates as a GeoJSON Point on the booking, and emits a \texttt{booking:new} Socket.io event to the room \texttt{worker\_\{workerId\}} so the worker sees an instant notification on their dashboard.

\texttt{updateBookingStatus()} implements the state machine. It uses a transition map to enforce valid status progressions (e.g., from \texttt{pending} only \texttt{accepted} or \texttt{cancelled} are valid). Each valid transition updates the \texttt{timeline} Map field with the current timestamp, increments the worker's \texttt{completedBookings} counter on completion, and emits a \texttt{booking:statusUpdate} Socket.io event to the room \texttt{booking\_\{bookingId\}}.

\subsection{workerController.ts}

\textbf{Purpose:} Manages worker profiles, the KYC application pipeline (which uploads documents to Cloudinary via Multer), availability toggles, location updates, and the earnings analytics aggregation. The \texttt{getEarnings()} function runs a MongoDB aggregation pipeline grouping completed, paid bookings by month:

\begin{lstlisting}[language=Java,caption={Earnings aggregation pipeline}]
const earnings = await Booking.aggregate([
  { $match: { worker: workerId,
              status: 'completed',
              paymentStatus: 'paid' }},
  { $group: {
      _id: { $month: '$createdAt' },
      total: { $sum: '$pricing' },
      count: { $sum: 1 }
  }},
  { $sort: { _id: 1 } }
]);
\end{lstlisting}

\subsection{matchingController.ts}

\textbf{Purpose:} Handles the worker search and ranking flow. It first queries MongoDB for all online, verified workers matching the requested profession within the specified service radius using a \texttt{\$near} geospatial query. The resulting candidate list is then forwarded to the AI Matching Engine via Axios with a 5-second timeout. If the engine is unreachable, the controller falls back to sorting by Haversine distance alone.

\subsection{adminController.ts}

\textbf{Purpose:} All admin-scoped operations. Includes functions for listing pending worker applications, approving or rejecting with notification email dispatch, fetching the executive dashboard metrics via a multi-collection \texttt{Promise.all} of aggregation queries, and the review moderation endpoints.

\section{Backend — Middleware}

\subsection{auth.ts (protect middleware)}

The \texttt{protect} middleware is applied to every authenticated route. It extracts the JWT from the request cookie (\texttt{req.cookies.token}) or the \texttt{Authorization: Bearer} header (for programmatic clients). It calls \texttt{jwt.verify()} with the secret; if the token is valid, it attaches the decoded user document from MongoDB to \texttt{req.user}. Any failure (missing token, expired token, tampered signature) returns a 401 immediately without calling the next handler.

\subsection{roleCheck.ts}

Provides role guard factory functions: \texttt{isAdmin}, \texttt{isUser}, and \texttt{isWorker}. Each reads \texttt{req.user.role} and calls \texttt{next()} if the role matches, or returns a 403 with a descriptive message otherwise.

\subsection{upload.ts}

Configures Multer with Cloudinary's storage adapter. File type is validated against an allowlist \texttt{(image/jpeg, image/png, image/webp)}; files outside this set are rejected with a 400 error before reaching the controller. Maximum file size is set to 5 MB.

\section{Backend — Services}

\subsection{razorpayService.ts}

Wraps the Razorpay SDK. Provides \texttt{createOrder()}, \texttt{verifyPaymentSignature()}, \texttt{getPaymentDetails()}, \texttt{refundPayment()}, and \texttt{capturePayment()} methods. The \texttt{verifyPaymentSignature()} method computes \texttt{HMAC-SHA256(orderId + ``|'' + paymentId, razorpaySecret)} and compares it against the signature sent by the client, implementing the standard Razorpay webhook verification protocol.

\section{AI Engine}

\subsection{matchingEngine.ts}

\textbf{Location:} \texttt{ai\_engine/src/algorithms/matchingEngine.ts}

Accepts a payload with \texttt{workers[]} (geospatial candidate list), \texttt{userLocation}, \texttt{urgency}, \texttt{skills[]}, \texttt{maxDistance}, and \texttt{minRating}. For each candidate worker it computes a \texttt{matchScore} using the weighted formula detailed in Chapter 8. Workers below \texttt{minRating} are filtered out. The result is sorted by descending \texttt{matchScore} and returned to the caller.

\section{Frontend — React Components}

\subsection{UserHome.tsx}

The main search and discovery page for authenticated users. It renders a two-panel layout: a \texttt{react-leaflet} MapContainer on the left showing all matched workers as custom coloured markers (green = high match, amber = medium, red = low), and a scrollable WorkerListPanel on the right. A search bar with debounced profession input, an urgency selector, a distance slider, and a minimum rating filter drive the \texttt{POST /api/matching/find} call on every change. Clicking a map marker or a list card opens the BookingModal.

\subsection{WorkerDashboard.tsx}

The worker's primary interface. It shows four KPI cards (Today's Bookings, Total Earnings, Average Rating, Completion Rate), an interactive earnings LineChart built with Recharts, a live booking requests table with Accept/Reject buttons, and the availability toggle. The dashboard subscribes to Socket.io events on mount and updates the pending bookings list in real time without page refresh.

\subsection{AdminDashboard (ExecutiveDashboard.tsx)}

A grid of six KPI cards plus two charts (monthly booking bar chart, category pie chart), all fed from \texttt{GET /api/admin/analytics}. \chapter{Worker Ranking Algorithm}
% ═════════════════════════════════════════════════════════════════════════════

\section{Overview}

The core intellectual contribution of UdhyogaPay is its multi-factor worker ranking algorithm. The goal is to surface the most relevant worker at the top of the results list rather than simply the nearest one. Simple proximity ranking fails for several reasons: a nearby worker with poor ratings or no relevant skills is worse than a slightly further worker who is excellent at exactly the required task.

\section{Factor Selection and Weights}

After analysing the literature on two-sided marketplace matching and conducting informal user interviews, the following three primary factors and weights were chosen.

\begin{table}[h!]
\centering
\caption{Ranking Score Factors and Weights}
\renewcommand{\arraystretch}{1.3}
\begin{tabular}{|p{3cm}|p{1.5cm}|p{9cm}|}
\hline
\thead{Factor} & \thead{Weight} & \thead{Rationale} \\
\hline
Distance & 50\% & Proximity is the dominant factor in hyperlocal services; a worker 1 km away is almost always preferred over one 20 km away holding all else equal. \\
\hline
Average Rating & 30\% & Rating captures historical quality signal. A 4.8-rated worker is significantly safer to book than a 3.2-rated one. \\
\hline
Experience & 20\% & Years of experience correlates with problem-solving ability and tool proficiency, particularly for complex tasks. \\
\hline
\end{tabular}
\end{table}

\section{Scoring Formula}

For each candidate worker $w$ and a user search with urgency $u$ and maximum distance $d_{max}$, the composite match score is computed as follows.

\textbf{Distance Score:}
$$D(w) = \left(1 - \frac{d_w}{d_{max}}\right) \times 100$$
where $d_w$ is the Haversine distance between the user's location and the worker's current location in kilometres, and $d_{max}$ is the user's specified maximum search distance. A worker at distance zero scores 100; a worker at exactly $d_{max}$ scores 0.

\textbf{Rating Score:}
$$R(w) = \frac{r_w}{5} \times 100$$
where $r_w$ is the worker's average rating on a 1--5 scale. A perfect 5.0 rating scores 100; a 2.5 rating scores 50.

\textbf{Experience Score:}
$$E(w) = \min\left(\frac{e_w}{10}, 1\right) \times 100$$
where $e_w$ is years of experience. The function is capped at 10 years (score 100) to prevent extreme outliers from dominating.

\textbf{Skill Match Bonus:} If the user submits specific skill requirements $S_{user}$, a skill overlap bonus is added:
$$\text{SkillBonus}(w) = \frac{|S_{user} \cap S_w|}{|S_{user}|} \times 10$$

\textbf{Urgency Multiplier:}
$$U(u) = \begin{cases} 1.0 & \text{if } u = \text{Low} \\ 1.2 & \text{if } u = \text{Medium} \\ 1.5 & \text{if } u = \text{High} \end{cases}$$

The urgency multiplier inflates the score for all workers equally when urgency is high, effectively amplifying the distance factor's dominance — the nearest available, highly-rated worker floats to the top in emergency situations.

\textbf{Final Composite Score:}
$$\text{MatchScore}(w) = \Bigl(0.50 \cdot D(w) + 0.30 \cdot R(w) + 0.20 \cdot E(w) + \text{SkillBonus}(w)\Bigr) \times U(u)$$

\section{Worked Example}

Consider a user in Hyderabad who searches for a plumber with urgency=High and maxDistance=10 km. Two candidate workers, A and B, have been returned by the MongoDB geospatial query.

\centering
\caption{Example Calculation Input}
\renewcommand{\arraystretch}{1.3}
\begin{tabular}{|p{3.5cm}|p{2.5cm}|p{2.5cm}|}
\hline
\thead{Attribute} & \thead{Worker A} & \thead{Worker B} \\
\hline
Distance to user & 2 km & 5 km \\
Average Rating & 3.5 / 5 & 4.8 / 5 \\
Experience & 2 years & 8 years \\
Skill overlap & None & Full match \\
\hline

\textbf{Worker A:}
$D(A) = (1 - 2/10) \times 100 = 80$
$R(A) = (3.5/5) \times 100 = 70$
$E(A) = (2/10) \times 100 = 20$
$\text{SkillBonus}(A) = 0$
$\text{Raw} = 0.50 \times 80 + 0.30 \times 70 + 0.20 \times 20 = 40 + 21 + 4 = 65$
$\text{MatchScore}(A) = 65 \times 1.5 = \mathbf{97.5}$

\textbf{Worker B:}
$D(B) = (1 - 5/10) \times 100 = 50$
$R(B) = (4.8/5) \times 100 = 96$
$E(B) = (8/10) \times 100 = 80$
$\text{SkillBonus}(B) = 10$
$\text{Raw} = 0.50 \times 50 + 0.30 \times 96 + 0.20 \times 80 + 10 = 25 + 28.8 + 16 + 10 = 79.8$
$\text{MatchScore}(B) = 79.8 \times 1.5 = \mathbf{119.7}$

Despite being further away, Worker B ranks higher due to superior rating, experience, and skill match. This outcome correctly reflects the user's best interest.

\section{Haversine Fallback}

The Haversine formula is used when the AI engine microservice is unavailable:
$$a = \sin^2\!\left(\frac{\Delta\phi}{2}\right) + \cos\phi_1 \cdot \cos\phi_2 \cdot \sin^2\!\left(\frac{\Delta\lambda}{2}\right)$$
$$d = 2R \arcsin\!\left(\sqrt{a}\right)$$
where $\phi$ is latitude, $\lambda$ is longitude, and $R = 6371$ km is the Earth's mean radius. Workers are sorted by ascending distance when the full scoring engine is unavailable.

\section{Authentication Security}

\subsection{JWT httpOnly Cookies}

Tokens are stored exclusively in httpOnly cookies, meaning JavaScript running in the browser cannot access them via \texttt{document.cookie}. This fully mitigates XSS-based token theft. The cookies are set with \texttt{Secure: true} in production (requiring HTTPS) and \texttt{SameSite: strict} to prevent CSRF attacks, as the cookie is only sent on same-site requests.

\subsection{Password Hashing}

All passwords are hashed using bcrypt with a cost factor of 12. At this cost, a single hash operation takes approximately 300 milliseconds on commodity hardware, making offline brute-force attacks economically unviable. The salt is concatenated with the hash and stored as a single string; no separate salt storage is required.

\subsection{Token Expiry and Rotation}

JWT tokens expire after 7 days. There is no refresh token mechanism in the current implementation; expired tokens require the user to re-authenticate. This is acceptable for a marketplace application where users return frequently.

\section{API Security}

\subsection{Rate Limiting}

The \texttt{express-rate-limit} middleware is applied globally with a window of 15 minutes and a maximum of 100 requests per IP address. The login endpoint has an additional stricter limit of 10 attempts per 15 minutes to prevent online brute-force attacks.

\subsection{Helmet.js}

Helmet.js sets 11 security-relevant HTTP response headers including:
\texttt{Content-Security-Policy}, \texttt{X-Frame-Options: DENY}, \texttt{X-Content-Type-Options: nosniff}, \texttt{Referrer-Policy: no-referrer}, and \texttt{Strict-Transport-Security} (in production). These headers directly address OWASP Top 10 categories A05 (Security Misconfiguration) and A03 (Injection).

\subsection{CORS Configuration}

CORS is configured to allow only the known frontend origin (\texttt{FRONTEND\_URL} environment variable). Credentials are allowed (\texttt{credentials: true}) to permit cookie transmission across origin. All other origins receive a CORS rejection before reaching any route handler.

\section{Data Security}

\subsection{MongoDB Injection Prevention}

All database queries use Mongoose's typed query API which parameterises all inputs. Raw string interpolation into query objects is never used. Additionally, the \texttt{express-mongo-sanitize} middleware strips any keys containing \texttt{\$} or \texttt{.} from request bodies, preventing NoSQL injection attacks where an attacker might submit \texttt{\{ "\$gt": "" \}} as a password to match any hash.

\subsection{File Upload Validation}

The Multer middleware configuration rejects any uploaded file whose MIME type is not in the explicit allowlist \texttt{(image/jpeg, image/png, image/webp)}. File size is capped at 5 MB. Files are uploaded directly to Cloudinary and never written to the local filesystem, eliminating the risk of path traversal attacks against local storage.

\subsection{Input Validation}

The Joi library validates all request bodies on sensitive endpoints (registration, booking creation, worker application). Any request body that fails validation returns a 400 with the specific validation errors; the handler is never invoked, preventing downstream processing of malformed data.

\section{Payment Security}

Razorpay payment verification uses HMAC-SHA256 signature verification. The server reconstructs the signature from \texttt{orderId|paymentId} using the \texttt{RAZORPAY\_KEY\_SECRET} and compares it against the client-submitted signature using Node.js's \texttt{crypto.timingSafeEqual()} to prevent timing side-channel attacks. Webhook events are similarly signature-verified before processing.

\chapter{Conclusion and Future Work}

\section{Summary}

UdhyogaPay represents a complete, production-grade full-stack web platform that successfully addresses the core problem of disconnection between local service workers and customers in the semi-urban Indian context. The project demonstrates that a four-person engineering team can, within a single academic year, design and deploy a marketplace platform that rivals the architectural complexity of commercial products while remaining fully open-source and deployable on free-tier cloud infrastructure.

The key technical achievements of this project are as follows. The multi-factor worker ranking algorithm, combining distance, rating, and experience with an urgency multiplier, demonstrably produces better search results than proximity-only ranking, as validated by the worked numerical example and by informal user testing. The real-time booking status propagation via Socket.io eliminates the need for polling and provides a seamless, app-like experience in the browser. The KYC verification pipeline provides a lightweight but effective gate for worker quality, directly addressing the trust deficit that plagues unorganised service marketplaces. The complete TypeScript codebase across frontend, backend, and AI engine ensures type safety and maintainability beyond the project submission.

\section{Limitations}

Several limitations should be acknowledged. First, the payment module, while fully code-complete with Razorpay integration, operates in demo mode because live payment processing requires a registered business entity with the payment gateway. Second, the platform currently seeds only six service categories; a production system would require a significantly larger and more granular category taxonomy. Third, the chat feature stores all messages in MongoDB without encryption at rest, which is a privacy limitation for sensitive job-related communications. Fourth, the platform has not been tested with MongoDB collections exceeding 10,000 documents; performance at scale requires further profiling and potentially the addition of a Redis caching layer.

\section{Future Enhancements}

Several directions for future enhancement are planned. The highest-priority enhancement is the integration of a live Razorpay payment flow with a registered business, including the escrow-like payment hold mechanism where payment is captured at booking creation and released to the worker's linked bank account upon service completion. This would transform UdhyogaPay from a booking platform into a complete financial intermediary.

The second priority is the launch of the React Native mobile application, which shares the same backend API and provides push notifications via Expo Notifications, background location tracking for workers, and biometric authentication (fingerprint/Face ID). The mobile codebase is already present in the repository and requires only API key configuration and EAS Build setup for App Store and Play Store submission.

Third, the matching algorithm should be enhanced with a machine learning model trained on historical booking outcomes. With sufficient data, a collaboratively filtered recommendation system can learn which worker-user pairings lead to five-star reviews and prioritise those combinations above what the hand-crafted scoring formula can achieve.

Fourth, a multi-language interface supporting Tamil, Telugu, Hindi, and Kannada would dramatically expand the addressable worker base, since many informal sector workers in South India are not comfortable in English.

Fifth, the introduction of subscription tiers for premium workers — who pay a monthly fee for priority listing, a verified badge, and access to advanced analytics — would create a sustainable revenue model beyond the commission-per-booking model.

\section{Learning Outcomes}

This project provided the team with hands-on experience across the full spectrum of modern web engineering: database schema design with geospatial indexes, REST API design with proper HTTP semantics, type-safe full-stack TypeScript development, real-time WebSocket architecture, cloud deployment with CI/CD, and the academic rigour of documenting a complex system to publication standard. The platform is submitted in a state of full deployment readiness and is publicly accessible, making it a demonstrable artefact of engineering capability.
